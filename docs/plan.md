# Дашборд для левого монитора (dwm)

Десктоп-виджеты на левом мониторе трёхмониторной системы под dwm:
фоторамка, часы, 3–4 окошка чатов Telegram, погода, строка иконок состояния системы.

---

## 1. Выбор стека

### Язык: Go

Отказ от Python — осознанный: PyGObject + venv + системные либы дают «зоопарк
версий». Go даёт:
- **один бинарь**, никакого рантайм-зоопарка;
- **горутины** — естественно ложатся на «несколько источников опрашиваются
  параллельно» (telegram-стрим, погода, /proc, раскладка);
- **`gotd/td`** — MTProto-клиент от имени пользователя (аналог Telethon),
  читает каналы/чаты, включая чужие.

Цена: GUI-код на cgo+GTK многословнее. Платим один раз на каркасе (M1).

### GUI-тулкит: пробуем GTK4 (`gotk4`), запасной — GTK3 (`gotk3`)

> ⚠️ Решение проверяется в M1. **Сначала пробуем GTK4**; если прибить окно к
> левому монитору под dwm на нём окажется неподъёмно — откатываемся на GTK3.

- **Первый заход: GTK4 через `github.com/diamondburned/gotk4`** — современные,
  активно развиваемые биндинги. Риск: GTK4 — Wayland-first, намеренно прячет
  X11, поэтому управление типом окна/стэкингом под dwm может быть труднее.
- **Запасной вариант: GTK3 через `github.com/gotk3/gotk3`** — даёт прямой доступ
  к `GdkX11` и предсказуемый `WM_CLASS` для правил dwm. В режиме поддержки
  (не развивается), но для десктоп-виджета этого достаточно.

### Зависимости (предварительно)

| Назначение | Библиотека / источник |
|---|---|
| GUI | `github.com/diamondburned/gotk4` (GTK4, cgo); запасной — `gotk3` (GTK3) |
| Telegram (user-client) | `github.com/gotd/td` (MTProto) |
| SVG-иконки | librsvg через GdkPixbuf (системная, идёт с GTK) |
| Погода | Open-Meteo (без ключа, JSON); запасной — `wttr.in` |
| Система | `/proc`, `sensors`, `setxkbmap -query`, `ip`/`wg` |

Системные пакеты (Arch): `gtk3`, `librsvg`, `pkgconf`, `go`.

## 2. Архитектура

Один процесс, один бинарь:

```
main
 ├─ deskwindow: по одному override-redirect окну на КАЖДЫЙ виджет
 │              (геометрия каждого — в config.toml; прозрачность при picom)
 ├─ goroutine: telegram client (gotd/td)  ──┐
 ├─ goroutine: weather poller             ──┤
 ├─ goroutine: system stats poller        ──┼─> каналы Go ─> UI
 ├─ goroutine: photo rotator              ──┤   (через glib.IdleAdd!)
 └─ goroutine: clock tick                 ──┘
```

**Ключевое правило GTK:** к виджетам прикасаемся только из главного потока.
Из горутин шлём данные через канал, обновляем через `glib.IdleAdd(...)`.
Единая дисциплина для всех виджетов.

Каждый виджет — отдельный пакет: `New() *gtk.Widget` + метод обновления,
дёргаемый из своей горутины. Стилизация — единый `style.css` через
`GtkCssProvider`; цвета/шрифты/пороги — в CSS, не в коде.

## 3. Структура проекта (предварительно)

```
dashboard/
├─ plan.md                 # этот файл — обзор майлстоунов
├─ M1/ … M7/               # по майлстоуну: plan.md (фазы) + progress.md
├─ go.mod
├─ main.go
├─ config.toml             # каталоги фото, контакты TG, координаты, пороги
├─ internal/
│  ├─ deskwindow/          # окно-десктоп под dwm (M1)
│  ├─ photo/  clock/  telegram/  weather/  sysstat/  ui/
├─ assets/icons/*.svg
└─ style.css
```

`config.toml` и session-файл Telegram — в `.gitignore`.

## 4. Майлстоуны

Каждый майлстоун живёт в своём каталоге `M{n}/` с двумя файлами:
- **`plan.md`** — фазы майлстоуна. *План.* Меняем только когда реально
  изменились планы.
- **`progress.md`** — *реальный прогресс.* Всегда отражает сделанное, даже если
  это расходится с планом.

| M | Каталог | Цель | Статус |
|---|---|---|---|
| **M1** | [`M1/`](M1/plan.md) | Каркас + окно-десктоп под dwm (критичный прототип всего стека) | ✅ завершён |
| **M2** | [`M2/`](M2/plan.md) | Часы + фоторамка (быстрый видимый результат) | ✅ завершён |
| **M3** | [`M3/`](M3/plan.md) | Строка иконок состояния системы | не начато |
| **M4** | [`M4/`](M4/plan.md) | Погода (Open-Meteo) | не начато |
| **M5** | [`M5/`](M5/plan.md) | Telegram: логин-сессия → poll → карточки | не начато |
| **M6** | [`M6/`](M6/plan.md) | Telegram: стриминг апдейтов, аватары, прокрутка | не начато |
| **M7** | [`M7/`](M7/plan.md) | Полировка: шрифты, кросс-фейд, плавность, автозапуск | не начато |

Порядок не жёсткий, но M1 — обязательно первым: пока окно не прибито к левому
монитору под dwm, остальное писать рано.
