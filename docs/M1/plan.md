# M1 — Каркас и окно-десктоп под dwm

**Цель:** доказать, что окно можно прибить к левому монитору как десктоп-виджет
под dwm, и заложить каркас приложения. Самый рискованный майлстоун — поэтому
первый. Закрывает разом весь стек: cgo, GTK, multi-head, дисциплину горутин.

> ⚠️ **Тулкит: GTK4 (`diamondburned/gotk4`) — подтверждён рабочим в Ph2.** Окно
> успешно прибивается к монитору под dwm через override-redirect. Откат на GTK3
> не понадобился.

> 🧩 **Модель окон (решено в Ph2): один виджет = одно окно.** Не одна панель с
> зонами, а N независимых override-redirect окон, каждое со своей `Geometry` в
> `config.toml`. Плюсы: легко добавлять/двигать виджеты, вплоть до переноса на
> другой монитор (это просто другие координаты). Каждое окно может быть
> **прозрачным** (при запущенном композиторе, picom) — для скруглённых углов и
> полупрозрачного фона. Без композитора окна непрозрачны (прозрачность = чёрное),
> деградируем мягко.
>
> 🖱 **Фокус:** override-redirect окна **получают события мыши** (клики, скролл,
> hover) — этого хватает для read-only Telegram с прокруткой. **Клавиатурный
> фокус по умолчанию НЕ дают** (dwm не маршрутизирует ввод в неуправляемое окно).
> Нашим виджетам клавиатура не нужна; если понадобится — отдельный `XSetInputFocus`.

## Фазы

### Ph1 — Пустое окно
- GTK4-окно без декораций (`SetDecorated(false)`).
- Запуск, чистое завершение.

### Ph2 — Размещение окна по координатам из конфига
Авто-определение монитора решено НЕ делать: GTK4 вообще не умеет позиционировать
окно само, а dwm для managed-окон сам диктует геометрию. Поэтому размещаем
напрямую через override-redirect + жёсткие координаты из `config.toml`.

- `config.toml`, секция `[window]`: `x`, `y`, `width`, `height` (левый монитор в
  общем X-пространстве обычно начинается с `x=0`).
- Окно делаем **override-redirect** — dwm полностью игнорирует такие окна
  (`manage()` проверяет `override_redirect`), так что и геометрию задаём мы сами.
- Координаты применяем на X11-уровне (`XMoveResizeWindow` / эквивалент через
  Gdk), т.к. сам GTK4 позиционировать не даёт.
- Критерий: окно появляется ровно в заданном прямоугольнике на левом мониторе,
  не тайлится, не ворует фокус, лежит под обычными окнами, остаётся на месте при
  переключении тегов/раскладок.

> 🔮 **Возможное улучшение на будущее (Ph2.1):** авто-определение геометрии
> левого монитора (`GdkMonitor` / `xrandr`), чтобы не править координаты руками
> при смене раскладки мониторов. Делаем, только если ручной конфиг станет неудобен.

### Ph3 — Каркас приложения (мультиоконный)
Переход от одного тестового окна к модели «виджет = окно».

- `config.toml`: список окон-виджетов, у каждого — тип виджета + своя
  `[window]`-геометрия (`x/y/width/height`). Загрузчик читает список и
  создаёт по окну на запись.
- Бэкенд `deskwindow` прибивает каждое окно (цикл `Pin()` по списку) — интерфейс
  это уже поддерживает.
- Прозрачность: включить RGBA-визуал и `background: transparent` через CSS;
  скруглённые углы — `border-radius` в CSS (видны при запущенном композиторе).
  Корректно деградировать к непрозрачному, если композитора нет.
- `style.css` через `GtkCssProvider` — общий для всех окон.
- В каждое окно — пустой заглушечный виджет (фото / часы / telegram / погода /
  иконки) под наполнение в M2–M6.

### Ph4 — Дисциплина обновлений
- Демонстрационная горутина → канал → `glib.IdleAdd` обновляет тестовый label.
- Это эталон, по которому потом пишутся все виджеты.

## Критерий готовности
Окно стабильно висит на левом мониторе, не мешает работе на других мониторах,
каркас собирается, демо-обновление из горутины работает.

- Закрыт вопрос «одно окно или несколько» → **одно окно на виджет** (Ph2).
